# JARVIS sensors 設計 (2026-07-07)

## 目的

「今のバックアップ状況は?」「マシンの調子は?」に対し、JARVIS が**聞かれた瞬間に実測**して
実測時刻つきで答える。体温計を当てるのはサーバ、モデルは検温結果を読み上げる係。

## スコープ / 非スコープ

- ✅ 読み取り専用センサー (backup / machines / storage)
- ❌ 書き込み系の手 (バックアップ起動等) — 将来やるなら人間確認+専用鍵+可逆操作限定が前提条件
- ❌ mini への常駐追加・新規ポート — 既存 SSH 鍵と既存ファイルだけ使う
- ❌ 定期キャッシュ主義 — 実測が本線。キャッシュ (朝9時 status チャンク) は実測失敗時の「最終既知値」のみ

## 安全不変条件 (このまま実装の受け入れ条件)

1. LLM が選べるのは `topic ∈ {backup, machines, storage, all}` の enum だけ。
   **LLM 由来の文字列がコマンドに混入する経路をつくらない**
2. 実行コマンドは人間が書いた固定リスト (下記 JSON) のみ。コードに秘匿情報
   (tailnet IP・ユーザー名・ホスト名) を書かない — **リポジトリは public**
3. ツールは読み取りのみ。タイムアウト必須 (コマンド8秒 / topic 全体12秒。
   timeout 時は start_new_session + killpg で shell の孫プロセスごと止める)
4. センサー系キーワードを含む質問には、モデルに任せずサーバが**応答生成の前に**
   必ず実測して結果を文脈注入する (決定的事前注入。モデルの tool-calling には依存しない)

## 構成

### 設定 (リポジトリ外, `~/Library/Application Support/Hisho/`)

- `backup_targets.json` (既存・変更なし) — topic=backup のコマンド台帳として再利用
- `sensor_targets.json` (新規) — topic=machines / storage のコマンド台帳。形は backup と同じ
  `{"topics": {"machines": [{"name":..., "cmd":...}], "storage": [...]}}`

### 新モジュール `core/hisho_core/sensors.py`

- 汎用実行係: `{name, cmd}` のリストを**並列**実行し、各行を
  `【name】stdout(または "実測失敗: 理由")` に整形して束ねる純関数 + subprocess 層
- `measure(topic, app_support_dir) -> str` : topic → 台帳選択 → 並列実測 → 実測時刻ヘッダ付き文字列
- 不達の機体はその行だけ「接続不可 (電源/tailnet を確認)」。他は正常に返す
- 台帳の形式不正エントリ (cmd 欠落・非dict・ルートが配列等) は実行せず
  「台帳エントリの形式が不正」の行として報告 (人間の編集ミスへの第二層防御)
- timeout は二層: コマンド 8 秒 (Popen start_new_session + killpg で孫ごと kill) と
  topic 全体 12 秒 (deadline 超過分は「実測失敗: 全体タイムアウト」)

### `tools.py` — `check_status` (REGISTRY のみ・tool 面は封印)

- REGISTRY に read-only 実装を登録。sensors の実測 + 全滅時の「最終既知値」引用を担う。
  **モデルには渡さない** (封印) — サーバの事前実測が REGISTRY 経由で呼ぶだけ
- TOOL_SPECS の定義は将来 tool-calling を再開する場合のために残すが、
  忘却ターンでモデルに渡す specs は forget_memories のみにフィルタする (M2 ゲート維持)
- 多層防御: ディスパッチ層でも「忘却ゲートを通っていないターンの forget_memories は
  実行拒否 + warning ログ」

### `server.py` — 決定的事前注入

- キーワードゲート: `状態|状況|調子|バックアップ|温度|容量|空き|稼働|生きて|落ちて|ディスク|動い`
  (forget と同じ位置に並置)
- ゲート発火時はサーバが**先に**実測し、レポートを追加 system メッセージ
  「以下はたった今この瞬間の実測データ (サーバが計測済み)。これだけに基づき平文で要約し、
  実測時刻を必ず添えること:」として文脈注入。モデルは要約のみで、**ツールは一切渡さない**
  (理由: 実LLMスモークで tool-calling 方式は語りが実測に先行し汚染される欠陥を確認。
  事後追記フォールバックも同欠陥のため廃止)
- topic 推定: バックアップ→backup / 温度・容量・空き・ディスク→storage /
  稼働・生きて・落ちて・マシン・動い→machines。ちょうど 1 群一致のみ採用、
  0 群 or 複数群一致は `all` (読み取り専用なので過剰測定が安全側)
- forget 意図との衝突: forget ゲートが先勝ち (既存挙動を変えない)
- センサー発話は RAG を両方向で遮断: memories 注入なし (古い状態記憶が新実測と矛盾して
  語りを汚染する。実測確認済み)、索引もなし (揮発性の実測値が「既知の事実」化するのを防ぐ)

### 応答規約 (persona 追記)

- 平文・記号なし (既存規約)。**「HH:MM 実測」を必ず添える**
- 実測が全滅した時のみ「実測できなかったため最終既知値 (定期収集分)」と明示して
  status チャンクを引用 (check_status が自動で付ける)
- 実測結果に無いことを推測で足さない (「盛らない」既存規約の再徹底)

## テスト

1. pytest: sensors 純関数 (整形/timeout/全体deadline/部分failure/台帳形式不正)、
   check_status (全滅フォールバック/不正topic の all 正規化)、server ゲート
   (事前実測→注入/topic 推定マトリクス/forget 先勝ち/specs フィルタ/ディスパッチ拒否/
   RAG 両方向スキップ/非発火/external 非発火) — 実行係・実測は fake 注入
2. **実LLMスモーク** (プロジェクト教訓: tool 挙動は実モデルでしか検証できない):
   別DB (`HISHO_DB`) + 別ポート (`HISHO_PORT`) で core を起動し curl で
   「今のバックアップ状況は?」「マシンの調子どう?」「モチの好物は?」(非発火確認) を実測
3. 完了条件: 実測値つき応答 / 全機不達でも応答継続 / 既存テスト green 維持
